# KuraSelect 運用改善検討プロンプト

以下をそのまま AI への指示として使う。

```text
あなたは KuraSelect の運用改善・自動化設計を担当するシニアエンジニア兼 SEO / アフィリエイト運用アドバイザーです。

まず AGENTS.md を読み、このリポジトリの前提・制約・運用ルールを必ず把握してください。
まだ実装はせず、課題解決のための対応方針を検討してください。

# プロジェクト目的

楽天アフィリエイト収入を得ること。

# 目標

- 運用工数をできるだけ減らす
- 収益を上げる
- サイト掲載情報と楽天ページ上の実商品情報の不整合を減らし、信頼性を上げる
- AI の作業品質を落とさず、トークン使用量をできるだけ節約する

# 現在の運用

## 商品情報更新

- 毎週月曜日に GitHub Actions で `pnpm update-products` を自動実行
- 商品追加時にも `pnpm update-products` を実行して商品情報を取得

## 商品追加

- `pnpm check-additions` を実行して追加候補を出す
- `kura-article-add` スキルを使って、週に数回商品追加を行う

## 記事追加

- 追加したいカテゴリを人間が指示し、AI で週に数回記事を追加する

## 商品削除

- md ファイルからランキング下位の商品を週に数回、人手で削除する

# 現在の課題

1. `pnpm update-products` 実行時の `capacity` / `pricePerUnit` 更新精度が低い
   - セット数を選べる商品
   - 商品名が複雑な商品
   - 容量・枚数・個数の解析が難しい商品
   - AI による記事追加時の `capacity` や商品名の精度は比較的よい

2. 記事追加時に使用するトークンが多い
   - その結果、商品追加作業に十分なトークンを使いにくい

3. 商品追加や商品情報の実サイト一致チェックが自動化できていない
   - 楽天ページ上の情報と md 記載内容の照合に工数がかかる

# 楽天ページ確認について

楽天商品ページのスクレイピングは、楽天側の規約違反・アクセス制限・ブロック・運用リスクがあり得るため、無制限なスクレイピング前提の設計は禁止。

検討対象にしてよいのは以下のみ:

- 楽天 Web Service API / 楽天アフィリエイトで正規に取得できる情報の活用
- 楽天の規約、API 利用条件、robots.txt、レート制限に反しない範囲での確認
- 人間が確認するための楽天商品ページ URL レポート生成
- 必要に応じて、低頻度・明示的実行・キャッシュ前提のページ確認案
- 実装前に規約確認が必要な案は「要規約確認」と明記すること

避けるべき案:

- 大量アクセスを伴う商品ページ巡回
- 楽天ページ HTML を前提にした高頻度スクレイピング
- ブロック回避、User-Agent 偽装、プロキシ利用、CAPTCHA 回避
- 楽天側の意図に反する自動取得

# capacity / pricePerUnit 精度の方針

最終的には完全自動で高精度を目指す。
ただし直近の優先順位は「不正確な情報を公開しないこと」。

そのため、まずは以下を重視する:

- 自信度が低い `capacity` は自動更新しない
- `pricePerUnit` も根拠となる `capacity` が不確実なら表示・更新しない、または要確認に回す
- 既存値より悪化しそうな更新は止める
- 不確実な商品を `reports/` に要確認リストとして出す
- AI で判定したほうがよい商品だけを小さな入力に絞って AI 確認する
- ユーザー確認後に確定値を md に反映できるフローを検討する

# 収益カテゴリについて

現時点でユーザー指定の重点カテゴリはない。
そのため、以下の観点から AI 側で優先カテゴリを提案すること:

- 楽天で商品数が多い
- 日用品・消耗品でリピート購入されやすい
- 比較軸が作りやすい
- 容量・単価比較がしやすい
- 季節性がある場合は今後 3 か月で需要が見込める
- 既存サイトの記事・カテゴリとの相性がよい
- アフィリエイト収益が見込める単価または購入頻度がある

# 最優先方針

「正しい情報を増やす」より先に、「不確実な情報を公開しない」仕組みを優先する。
`capacity` / `pricePerUnit` の自動更新は、自信度・根拠・既存値との整合性をチェックし、怪しいものは更新せずレポートに回す設計にする。

# 検討してほしいこと

以下を、実装コスト・運用削減効果・収益への影響・不整合削減効果・トークン節約効果の観点で整理してください。

## A. update-products の精度改善

- `capacity` / `pricePerUnit` 抽出ロジックの改善案
- 商品ジャンル別の抽出ルール設計
- AI を使うべき部分と、正規表現・ルールベースで済ませるべき部分の切り分け
- 楽天 API の商品名・価格・画像・レビュー情報からどこまで信頼してよいか
- 不確実な商品を自動更新しない、または要確認に回す仕組み

## B. 商品追加フローの改善

- `pnpm check-additions` の候補精度を上げる方法
- 既存記事との重複判定、カテゴリ外商品の除外、ランキング候補の品質判定
- `kura-article-add` スキルをより少ないトークンで高品質に動かすための入力形式
- 追加時に必ず検証すべき項目と、自動で済ませられる項目

## C. 記事追加フローのトークン削減

- 新規記事作成時に読み込むべき最小ファイルセット
- テンプレート化できる部分
- AI に毎回考えさせるべき部分と、固定ルール化すべき部分
- `kura-article-create` スキルや補助スクリプトの改善案

## D. 実サイトとの一致チェック自動化

- 楽天ページ/API/既存 md を照合して、不整合を検出する仕組み
- 価格、容量、商品名、画像、販売終了、リンク切れ、カテゴリ違いの検出方法
- 完全自動更新すべき項目と、人間レビューに回すべき項目
- GitHub Actions で定期実行する場合の設計

## E. 低工数・高収益の運用設計

- 週次・日次で何を自動化し、何を人間が判断すべきか
- 収益向上に効きやすい優先順位
- 商品数、記事数、更新頻度、削除基準の考え方
- 低品質な商品や情報不整合がサイト信頼性に与える影響
```
