# KuraSelect スクリプト実装計画書スキル

## Workflow

1. 対象 script・lib・tests・package.json を必要な範囲だけ Read する。類似の既存計画書が明示された場合のみ `docs/` も確認する。
2. `docs/IMPLEMENTATION_PLAN_{FEATURE}.md` を作成する。
3. 計画書に目的・変更方針・変更ファイルと変更箇所・テスト・実装手順・確認コマンドを書く。
4. Self Review を実施し、問題があれば計画書をその場で修正する。

---

## Plan Requirements

- **既存関数の変更**: 変更前/変更後のコードスニペットを両方書く
- **新規ファイル**: export する API・責務・呼び出し元の変更を書く（変更前は不要）
- **状態遷移・条件分岐**: テーブルで整理する
- **関数名・セクション名**: 必ず明記する。行番号は可能な範囲で添える
- **確認コマンド**: `package.json` の scripts を Read して存在するものだけ書く。`pnpm test` は基本、`pnpm build` は記事・スキーマ・フロント影響時に追加
- **非対象**: 影響しない記事本文・スキーマ・デプロイ設定は計画に含めない

---

## Self Review

計画書を書いたら以下を1項目ずつ確認し、問題があれば計画書を修正する。

- **スコープ**: ループ内で宣言した変数をループ外で参照していないか
- **正規化の一貫性**: 比較ロジックの両辺が同じ正規化パイプラインを通しているか
- **型と実データ**: frontmatter / YAML パース結果の実データと型定義がズレていないか（空文字 vs null など）
- **空値の明示性**: 空文字・null・undefined の扱いが暗黙の挙動に依存せず明示されているか
- **テストの直接性**: テストが実装関数を直接 import しているか（複製ヘルパーになっていないか）
- **import/export の過不足**: 未使用の import・必要な export の漏れがないか
- **責務の分離**: 状態更新・候補採用・確定判定が混在していないか

---

## Completion

ユーザーには簡潔に伝える。

- 作成した計画書のパス
- Self Review で直した点（なければ「指摘なし」）
- 実装に進む場合の推奨 Step
