# KuraSelect 実装計画レビュー

実装計画書を、実装前に破綻しないか確認する。長い説明よりも、現行コードと噛み合わない点を短く指摘する。

## 読むもの

1. 対象の実装計画書。
2. 計画書が変更対象として挙げているファイル。
3. 関連テスト、`package.json` の script、既存 helper や型定義。
4. 再レビューの場合は、前回 Finding が解消されているか分かる範囲。

必要な範囲だけ読む。`node_modules`、`dist`、`.astro` は読まない。

## レビュー観点

- 計画の目的、変更範囲、実装手順が現行コードと一致しているか。
- import/export、スコープ、型、ファイル拡張子、関数名、script 名に実装不能な不整合がないか。
- 新規 helper の切り出しがテスト可能性を上げているか。テスト側にロジック複製が残るなら import に寄せられないか。
- 既存のガード、`review` / `rejected` / `matched` などの状態遷移を壊さないか。
- dry-run と write 実行の違い、`.bak` 生成、記事 frontmatter、Zod スキーマへの影響を見落としていないか。
- 確認コマンドが実在し、KuraSelect の `pnpm` 前提に合っているか。
- テストが変更の本体を直接検証しているか。単なる複製 helper のテストになっていないか。

## 返答形式

Findings first で返す。重大なものから並べ、可能なら対象ファイルへのリンクを付ける。

- blocking: 実装すると壊れる、テストできない、計画と現行コードが矛盾している。
- non-blocking: 品質・保守性・説明粒度の改善。

問題がなければ、重大・中程度の問題がないことを明言し、実装に進めてよいと短く伝える。

## 再レビュー

前回指摘が反映されたかを最優先で見る。解消済みの内容を長く再説明しない。残った矛盾、新しく生まれた不整合、過剰に複雑になった箇所だけ指摘する。
